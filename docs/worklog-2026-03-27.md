# 2026-03-27 Worklog

## 1. 实现 worker 侧 Chrome HTTP 代理路由

为 worker 新增了一组统一的 Chrome HTTP 代理入口，按 `accountIdx` 路由到对应 profile 的 debugger port：

- `GET /chrome/:accountIdx/json/version`
- `GET /chrome/:accountIdx/json/list`
- `POST /chrome/:accountIdx/json/activate/:targetId`
- `POST /chrome/:accountIdx/cdp/call`

特点：
- 挂在 worker HTTP 服务下
- 复用现有 `authMiddleware`
- 不影响现有 `/rpc/:toolName` 调用方式
- 首版只支持 HTTP，不支持 websocket upgrade

## 2. 抽出共享端口解析逻辑

新增共享 resolver，统一 `accountIdx -> debuggerPort` 的解析优先级：

1. 优先取 `~/Private/chrome.json` 中 `account_<idx>.port`
2. 否则回退到 `runtime-registry` 中的 `debuggerPort`
3. 都没有则报错

这样保证：
- `/chrome/*` 路由
- `/rpc/chrome_get_targets`
- `/rpc/chrome_cdp_call`

三者对同一账号使用同一套端口解析规则。

## 3. 复用现有 CDP helper

没有新造协议层，直接复用已有 helper：

- `getVersion`
- `getTargets`
- `activateTarget`
- `callCdp`

本次实现本质上是对现有 CDP helper 的 HTTP facade。

## 4. 保持首版边界清晰

本次明确没有做：

- websocket 代理
- `server.on("upgrade")`
- 双向 websocket 转发
- `webSocketDebuggerUrl` 改写

原因：当前 worker 代码里没有 websocket proxy 基础设施，先完成 HTTP 最小闭环更稳妥。
